1. Vision & Principles

Product Vision

One-liner: Kitab is a comprehensive habit and activity tracker that helps users build and maintain good habits, grow as individuals and as a community, and discover powerful insights from their data.

Core metaphor: Your life is a book being written in real-time. Every habit you build, every action you take, every streak you maintain is a page in your Kitab. The app makes that book visible, measurable, and beautiful.

Marketing tagline: "Track your journey. Grow every day." — The Islamic inspiration lives in the design system and optional features (Hijri calendar, prayer times, Islamic greetings), not in public-facing taglines. The app is inclusive to all users.

Cultural foundation: The word "Kitab" comes from the Arabic word for "book." The app draws inspiration from the Islamic concept of the book of deeds — a record of one's actions presented on the Day of Judgement. Kitab acts as a digital, living version of that concept. While rooted in Islamic philosophy, the app is inclusive and equally valuable to all users regardless of background.

Design Principles (Ranked)

When two principles conflict, the higher-ranked one wins.

RankPrincipleDescription
1Clarity over clevernessEvery screen should be immediately understandable. No mystery icons, no hidden gestures, no jargon. A first-time user and a power user should both feel at home.
2Progressive depthSimple on the surface, powerful underneath. A beginner sees a clean tracker. A power user discovers custom categories, detailed analytics, and automation. The complexity is there but never forced.
3Reflection, not judgmentThe app feels like a wise companion, not a disappointed parent. Missed a habit? Here's your pattern. Not a guilt trip. The tone is encouraging, insightful, and never punishing.
4Beauty with purposeEvery visual element earns its place. Islamic geometric patterns represent the interconnectedness of habits and growth. Gold accents mark achievement and progress. Nothing is purely ornamental.
5Accessible by defaultNot an afterthought. Every color, every interaction, every piece of text is designed to work for everyone — screen readers, color blindness, motor impairments, large text. If it doesn't work for everyone, it doesn't ship.
6Offline is realOn native devices, the app works fully without internet. Sync is seamless and invisible. Users should never think about connectivity.

2. Features & Information Architecture

Core Concept: Activities

Activities are the fundamental unit of Kitab. Everything the user tracks is an activity. An activity has a template (configuration) and entries (logged instances).

Activity Template (configuration)
├── Identity: name, category (icon + color), description
├── Fields: what data to capture per entry (13 types)
├── Schedule (optional): when the activity is expected
│   ├── Calendar type (Gregorian / Hijri)
│   ├── Frequency (daily, weekly, monthly, yearly, custom)
│   └── Time window (specific or dynamic prayer times)
└── Goals (optional): targets to achieve
    ├── Quick Setup ("do this 3x per week")
    └── Custom Goal (X comparison Y + consistency layer)

Activity Entry (logged instance)
├── Linked to template (Layer 1) — or ad-hoc
├── Linked to period (Layer 2) — or orphaned
├── Field values (captured data)
├── Notes (freeform text)
├── Timer segments (if timed)
└── Source (manual or V2 health device)

A "habit" is not a separate concept — it is an activity with a frequency schedule and a boolean goal (did I do it or not). The word "habit" may appear in the UI but the underlying model is always an activity.

Key Feature Areas

Streaks

Streaks count consecutive goal successes. Three levels:

LevelWhat It TracksDriven By
Per-activity streakDid I do this activity?activity_period_statuses
Per-goal streakDid I meet this specific goal?goal_period_statuses
All-goals day streakDid I meet all daily goals today?Aggregate of above (daily only)

Icons: 🔥 active streak, 🧊 frozen (pending items exist). Milestones at: 7, 14, 21, 30, 60, 90, 100, 180, 365 days.

Conditions (Life Events)

Conditions are life markers (illness, travel, injury, etc.) that span entire days. They exist independently from activities — conditions record when something happened, and activities can be excused with a condition as the reason.

  • Day-level granularity (start date, end date)
  • Non-overlapping per preset type (can't have two "Sick" conditions overlapping)
  • Show as chips on Home screen when active
  • Interwoven in Book screen day headers
  • Preset types managed in Settings (Sick, Traveling, Injured, Rest Day, Menstrual, Bereavement + custom)
Routines (Habit Stacking)

A routine is a sequence of activity templates grouped for habit stacking. The user starts the routine and flows through activities one by one. See §6 Routines for full specification.

Social Features

Friends, selective sharing (per-activity, per-friend), emoji reactions (5 emojis + 5 canned messages), and competitions. No public feed, no posting, no free-text messaging. See Social Screen for full specification.

Competitions

Users create or join challenges. Public or private. Required start and end dates (max 3 months). One or more leaderboards per competition, each tracking a different metric. Individual only (teams = V2). See Social Screen for details.

12. Activity Templates

The activity template is the foundational data structure. Every entry, goal, streak, and insight depends on how activities are configured.

Terminology

TechnicalUser-Facing
Activity TemplateActivity
Activity Log/RecordEntry
MetricField
ExcuseReason (non-judgmental)
Frequency PeriodNot exposed (contextual: "today", "this week")

Template Sections

1. Details

  • Activity Name: Required. Unique per user (case-insensitive).
  • Category: Required. Provides icon + color (no separate icon/color per activity).
  • Description: Optional.

2. Schedule (toggle, default OFF)

When ON: Calendar type (Gregorian/Hijri), start date, optional end date (with ✕ clear button), repeat frequency, expected entries (once/multiple per period), optional time window (specific or dynamic times).

Repeat options: Daily, Weekly (specific days), Monthly (specific days), Yearly (specific days/range within a specific month — works with both Gregorian and Hijri calendars, e.g., Ramadan 1-29/30, or January 1-31), Custom Interval (every X days/weeks/months/years).

Dynamic times from the Aladhan API: Fajr, Sunrise, Duha, Dhuhr, Asr, Maghrib/Sunset, Isha, 1/3 Night, Middle of Night, 2/3 Night. Both start and end must be same type (no mixing). Optional offset in minutes: e.g., Sunrise +15 min for Duha, or Fajr -30 min. Midnight crossover warning if end ≤ start.

High-latitude locations: If sunrise/sunset unavailable, dynamic times fall back to full-day period. Additional setting for adjustment method (Middle of Night, One-Seventh, Angle-Based, Nearest Latitude).

Schedule requires goals. If schedule is ON but no goals defined, system auto-creates an implicit habit goal ("Did at least one entry get logged?").

3. Fields (1–20 per template)

13 field types. All optional to fill in during logging (no required fields). See Component Library for the full field type table. Default Notes field always present on every entry (not configurable). Fields ordered by template sort order, reorderable via drag.

4. Goals (toggle, default OFF)

Quick Setup: Sentence builder. With schedule: "I want the [field] to be [at least ▾] [5] [km] per [day]" (period locked to schedule). Without schedule: "I want the [weight] to be [at most ▾] [200] [lbs]" (no period — standing target).

Comparison options: ≥, ≤, >, <, =, between, not between.

Custom Goal: X [comparison] Y formula builder. X = field + scope (most recent, current period, last N entries/days, all-time) + aggregation (sum, count, lowest, highest, average, mode). Y = static value or dynamic (another aggregation). Optional consistency layer: "meet this goal in at least X of the last Y periods."

Primary goal: One goal designated as primary. Shown on Home screen card. Drives all-goals day streak. Other goals have their own per-goal streaks.

5. Summary View

Auto-generated plain text summary at the bottom of template setup. Updates live as user configures.

Schedule Versioning

When schedule/goals change, user is asked: future only (new version appended, old periods preserved) or retroactive (versions collapsed, all statuses recomputed). Smart detection: if applying future-only but start date is in the past, auto-adjust to today.

16. Period Engine

Computes when activities are expected. Periods are computed on-the-fly, never stored in the database.

Why Not Stored?

  • Open-ended schedules = infinite rows
  • Dynamic time windows depend on real-time location (travel changes prayer times)
  • Schedule versioning requires recomputation

Computation by Frequency

FrequencyPeriod SpanNotes
DailyFull day (midnight-midnight Gregorian, sunset-sunset Hijri)With time window: narrowed
WeeklySelected days or full weekConsecutive days: one period or separate (user choice). Week start from Settings.
MonthlySelected days of monthConsecutive: one or separate. Short month: defaults to last day (not skip).
YearlySpecific days within a specific monthWorks with Gregorian or Hijri. E.g., Ramadan 1-29/30, Dhul Hijjah 1-9.
CustomSingle day per recurrenceEvery X days/weeks/months/years from start date.

When Computed

On app open, Home navigation, midnight/sunset boundary, activity detail view, streak calculation, entry linkage. Never precomputed for future dates. Pure math — fast (milliseconds for 365 days).

"Today" Definition

Always midnight-to-midnight for Home screen and all-goals streak, regardless of activity's calendar type. Hijri activities still show if their period overlaps.

17. Goal Engine

Evaluates whether goals are met. Manages active vs finalized evaluation and caching.

Active vs Finalized

StateDisplayStorage
Active period (ongoing)"In progress" (neutral, not red)Computed on-the-fly, cached in memory
Finalized period (ended)Met / Not met / ExcusedWritten to goal_period_statuses

Goal Types

Frequency (count entries), Value (aggregate field values), Custom (X comparison Y with dynamic targets), Consistency (pass rate across N periods), Combined (AND/OR groups). Full evaluation logic for each type in SPEC.md §10.3.

Goal-Level Reasons

When a period is completed but a specific goal fails, the user can add a reason per goal (not just per period). If an active condition exists, the app proactively prompts: "Would you like to add a reason? [Skip] [Add Reason]"

Caching

In-memory (Riverpod state). Invalidated by: entry changes, goal config changes, schedule changes, period status changes, sync pulling new data. Finalized periods use DB rows as permanent cache.

18. Linkage Engine

Connects entries to templates and periods. The bridge between what users log and the structure they defined.

Two-Layer Linkage

LayerWhatOptions
Layer 1Activity TemplateLink, change, unlink (becomes quick entry)
Layer 2Period (within template)Link, change, unlink. Only if Layer 1 set + template has schedule.

Linkage Timestamp Priority

For auto-suggesting which period: 1) Start Time field, 2) End Time field, 3) logged_at. Always overridable by user.

Frozen Period Boundaries

When linked, period_start and period_end are frozen on the entry (UTC). Handles timezone changes and travel. Never recalculated unless user chooses "apply retroactively" on schedule change.

Pending Detection

On app open: for each scheduled activity, check past periods with no entry + no status → create 'pending' row in activity_period_statuses. Makes "Needs Attention" queries simple DB lookups.

19. Sync Engine

Architecture

Native App → writes to → Local SQLite (Drift)
           → queues in → sync_queue table
           → when online → pushes to Supabase
           → pulls latest → merges into Local SQLite

Web app reads/writes Supabase directly (no local DB).

Principles

  1. Local-first on native: Every write → local first. UI reads local.
  2. Cloud-only on web: Direct Supabase. Offline = "You're offline" message.
  3. Last-write-wins: Most recent updated_at wins per record.

Sync Triggers

TriggerAction
App openFull sync (push + pull)
Every 5 minutesBackground sync
Home navigationSync if > 1 min since last
Significant writePush immediately
Pull-to-refreshFull manual sync
Foreground resumeSync if > 1 min since last

Syncs run on background thread. Never block UI. If sync in progress + another triggered, second is queued.

23. Sync Conflict Resolution

12-Step Merge Algorithm

Step 1:  Merge settings/profile (last-write-wins)
Step 2:  Merge categories (name conflicts → last-write-wins, cascade IDs)
Step 3:  Merge condition presets (label conflicts → last-write-wins, cascade IDs)
Step 4:  Merge activities (name conflicts → ASK USER, cascade entries)
Step 5:  Merge condition records (overlap → merge dates, cascade IDs)
Step 6:  Merge entries (union + last-write-wins on same ID)
Step 7:  Recompute period statuses from entries (preserve user decisions)
Step 8:  Recompute goal statuses from entries (preserve user excuses)
Step 9:  Merge friends (most advanced status wins)
Step 10: Merge activity shares (cascade to winning template IDs)
Step 11: Merge competitions + participants + entries
Step 12: Clear and regenerate notifications

Steps 7-8 are recomputation, not merge — system-computed statuses (completed, pending, met, not_met) are recomputed from merged entries. User-set statuses (excused, missed) are preserved.

Template Conflict UI

When duplicate activity names detected, sync pauses. User sees both versions: schedule summary, field/goal count, entry count, last edited date, last record date. User picks which to keep. All entries from loser relink to winner.

25. Security & Privacy

Row Level Security (RLS)

All Supabase tables have RLS enabled. Users can only access their own records (user_id = auth.uid()). Social tables have expanded policies for accepted friends. Competitions readable by participants.

Private Activities

Activities/routines can be marked Private (is_private boolean). Name and details blurred throughout the app. Tap to reveal temporarily. Double-tap "Kitab" title on Home to reveal all (toasts "Private activities revealed"). Auto-re-blurs on app background. Push notifications use generic text. Private activities CAN be shared — privacy is about onlookers, not friends. Recipients can independently mark shared items as private on their device.

Other Security

  • Input validation: Max lengths per field, sanitized text, no HTML rendering.
  • Tokens: JWT in secure storage (native), memory (web). Auto-refresh.
  • Encryption: TLS in transit, AES-256 at rest (Supabase). Local SQLite unencrypted V1 (device security boundary).
  • JSONB safety: 100KB max payload, strict types, no string concatenation.
  • Rate limiting: Supabase built-in for auth. 100 records/sync batch.
  • No secrets in client code. Service role key only in Edge Functions.
